## 2. 第一阶段设计结论

第一阶段音乐模块必须同时满足以下目标：

- 真实接入并验证 4 个播放器：
  - QQ 音乐
  - 网易云音乐
  - 酷狗音乐
  - 汽水音乐
- 默认模块保持为音乐模块
- 展开态提供真实可用的音乐内容页，而不是占位视图
- 收起态提供真实播放器来源与播放指示
- 权限失败时进入明确引导态，而不是静默失败
- 不引入冻结契约变更
- 不把 Apple Music / Spotify 写成第一阶段已支持

第一阶段的控制能力下限固定为：

- 启动播放器
- 读取当前来源
- 读取封面 / 歌名 / 歌手 / 时长
- 播放 / 暂停
- 上一首 / 下一首

以下能力明确不进入第一阶段：

- 进度拖拽
- 本地时间轴纠偏
- Apple Music 正式支持
- Spotify 正式支持

## 3. 第一阶段范围

### 3.1 In Scope

- 音乐模块 expanded UI 正式实现
- 4 个首期播放器的真实读取与控制接入
- 空态、播放态、暂停态、启动中、权限失败态、未安装态、控制失败态、非首期播放器态
- 收起态播放器来源 Logo / 播放指示
- 活跃播放器仲裁逻辑
- 低频关闭态会话探测
- 单元测试与必要的人工验证矩阵

### 3.2 Out Of Scope

- Apple Music / Spotify 第一阶段正式接入
- 进度拖拽
- 高精度本地时间轴推进与 seek 校准
- 模块线程直接修改：
  - `OverlayState`
  - `NotchModuleContext`
  - `ModuleEnergyPolicy`
  - 多屏窗口结构
  - Shell 顶层生命周期契约
- 模块自行创建完整 `NSPanel`
- 关闭态高频轮询

### 3.3 首期播放器入口规则

第一阶段空态只展示这 4 个已验证目标：

- QQ 音乐
- 网易云音乐
- 酷狗音乐
- 汽水音乐

Apple Music 与 Spotify 在第一阶段：

- 不出现在空态入口中
- 不写成“已支持”
- 如果当前活跃会话恰好来自这两个播放器，则模块进入“当前播放器不在第一阶段支持范围内”的诚实降级态

## 4. 设计输入确认

本次设计已经与需求方确认以下决策：

- 第一阶段必须包含真实播放器读取与控制接入
- 首期只要求 4 个已验证播放器真实可用
- Apple Music / Spotify 继续保留 `target`
- 空态入口第一阶段只展示 4 个首期播放器
- 控制能力中，除进度拖拽外，其余核心控制都属于第一阶段必做
- 收起态播放器指示属于第一阶段必做范围
- 权限失败时需要明确失败态与授权引导

---

## 5. 推荐实现路线

第一阶段采用“模块内聚 + Shell 最小接线”路线。

即：

- 音乐模块内部承担播放器适配、会话聚合、展示状态生成
- Shell 只新增最小范围的收起态摘要消费接线
- 不修改冻结契约
- 不把音乐能力过早抽成新的跨模块共享核心

采用此路线的原因：

- 能尽快落地 4 个首期播放器
- 风险集中在音乐模块内部，便于验证与回退
- 可以满足收起态播放指示的产品要求
- 能避免为了结构“更漂亮”而过早触碰冻结边界

## 6. 架构设计

第一阶段音乐模块按 4 层组织，先全部落在 `Modules/Music` 目录下。

### 6.1 Player Adapter 层

职责：

- 识别目标播放器是否安装
- 启动目标播放器
- 读取当前元数据
- 执行播放 / 暂停 / 上一首 / 下一首
- 对外声明该播放器当前在第一阶段的可用能力

第一阶段至少包含 4 个适配器：

- `QQMusicAdapter`
- `NeteaseMusicAdapter`
- `KugouMusicAdapter`
- `QishuiMusicAdapter`

这一层屏蔽播放器差异，不把控制细节散落到 UI。

### 6.2 Playback Session / Resolver 层

职责：

- 把多个播放器返回的快照归一化为统一会话
- 处理活跃播放器仲裁
- 表达当前播放源、歌曲元数据、控制能力、权限状态、错误状态

这层输出的是统一的 `PlaybackSession`，不是某个播放器的原始结构。

### 6.3 Music Presentation Store 层

职责：

- 把统一会话映射为 expanded 与 collapsed 所需的展示状态
- 管理用户操作过程中的短时状态：
  - `launching`
  - `controlPending`
  - `controlFailed`
- 管理失败态与权限态文案
- 管理收起态摘要输出

### 6.4 View 层

职责：

- `MusicModuleView` 承担 expanded 态 UI
- Shell 收起态只消费轻量 `CollapsedMusicSummary`

View 层不直接接触播放器控制细节。

## 7. 数据模型

### 7.1 Player Capability

需要显式声明播放器能力，不允许 UI 默认假设所有播放器都具备相同能力。

建议模型字段至少包括：

- `bundleID`
- `displayName`
- `isInstalled`
- `canLaunch`
- `canReadMetadata`
- `canPlayPause`
- `canSkip`
- `phase`

其中 `phase` 在第一阶段只需要表达：

- `v1Supported`
- `target`
- `unsupported`

### 7.2 Playback Snapshot

适配器返回的统一快照至少包括：

- `bundleID`
- `displayName`
- `isRunning`
- `isPlaying`
- `title`
- `artist`
- `artwork`
- `duration`
- `elapsedTime`
- `controlAvailability`
- `permissionState`
- `errorState`
- `capturedAt`

### 7.3 Playback Session

会话层统一聚合后的结构至少包括：

- 当前活跃播放器来源
- 标题 / 歌手 / 封面 / 时长
- 当前播放状态
- 当前播放器第一阶段是否受支持
- 当前控制是否可用
- 当前权限是否满足
- 最近一次活跃时间

### 7.4 Collapsed Music Summary

供 Shell 收起态消费的摘要结构保持最小化，只包含：

- 播放器来源 Logo 或来源标识
- 是否处于播放中
- 必要时的短文本提示

这一层不承担完整元数据 UI。

## 8. 状态模型

### 8.1 Expanded 态

第一阶段 expanded 音乐模块至少包含以下状态：

- `empty`
  - 当前没有可识别的活跃播放会话
  - 展示 4 个首期播放器入口
- `launchingPlayer`
  - 用户点击入口后，正在尝试启动播放器
- `playing`
  - 当前识别到首期支持播放器且正在播放
- `paused`
  - 当前识别到首期支持播放器但处于暂停
- `permissionRequired`
  - 当前读取或控制链路所需权限未满足
- `playerNotInstalled`
  - 用户点击了未安装的首期播放器入口
- `launchFailed`
  - 播放器已安装，但启动失败
- `controlFailed`
  - 控制命令发出后未得到确认
- `unsupportedActivePlayer`
  - 当前活跃播放器不在第一阶段支持范围内
- `metadataUnavailable`
  - 播放器运行中，但暂时无法读取到完整元数据

### 8.2 Collapsed 态

第一阶段 collapsed 只定义两类结果：

- `idle`
  - 没有活跃播放会话，或当前不需要展示音乐摘要
- `playingSummary`
  - 当前存在首期支持播放器且正在播放

第一阶段不单独做 `pausedSummary`，避免在收起态传达模糊的“仍然活跃”语义。

---

## 9. 活跃播放器仲裁规则

第一阶段不能简单使用“最后一个返回快照的播放器”，必须有稳定仲裁规则。

建议规则固定如下：

1. 如果只有一个首期支持播放器处于 `playing`，直接选它
2. 如果多个首期支持播放器同时处于 `playing`，优先最近一次成功控制或最近一次活跃来源
3. 如果没有 `playing`，但存在可识别暂停会话，选最近一次活跃播放器
4. 如果当前系统会话来源不是首期 4 个播放器，但能识别来源，则进入 `unsupportedActivePlayer`
5. 如果没有任何可识别会话，进入 `empty`

目标是避免 UI 在多个播放器之间频繁跳源。

## 10. 用户操作流

### 10.1 空态启动

用户点击空态播放器入口后：

1. 模块立刻进入 `launchingPlayer`
2. 对应适配器尝试启动播放器
3. 进入短时会话建立等待
4. 若建立成功：
   - 有活跃播放则进入 `playing`
   - 有可识别暂停会话则进入 `paused`
5. 若超时未建立成功：
   - 回退到 `launchFailed`
   - 再允许用户返回空态重试

### 10.2 播放 / 暂停

用户点击播放或暂停后：

1. Presentation Store 记录一次控制请求
2. 适配器发出控制命令
3. 下一轮真实快照确认是否成功
4. 成功则进入对应真实状态
5. 失败则进入 `controlFailed`

### 10.3 上一首 / 下一首

流程与播放 / 暂停一致，但确认结果以新快照变化为准。

### 10.4 收起态展开

收起态点击入口展开后：

- 默认仍进入音乐模块
- 优先展示当前最新音乐状态
- 不做额外模块切换

## 11. UI 结构设计

第一阶段 expanded UI 必须遵循已确认的 Figma 结构，不允许重新发明页面结构。

固定结构分区：

- `Tab`
- `设置`
- `内容`
- `音乐信息`
- `操作模块`
- `进度模块`

### 11.1 Empty

内容区展示：

- 引导文案：“美好的一天，从音乐开始”
- 4 个首期播放器入口

不展示：

- Apple Music
- Spotify

### 11.2 Playing / Paused

内容区展示：

- 封面
- 歌名
- 歌手
- 时长信息
- 播放 / 暂停按钮
- 上一首 / 下一首按钮

第一阶段允许展示读取到的进度与总时长，但不承诺拖拽，不承诺时间轴校准。

### 11.3 Permission Required

内容区展示：

- 当前缺失的权限类型
- 需要它的原因
- 去系统哪里开启
- 重试入口

### 11.4 Unsupported Active Player

内容区展示：

- 当前播放器不在第一阶段支持范围内
- 明确说明首期支持的 4 个播放器范围

## 12. 权限与失败策略

第一阶段不能把权限问题隐藏为“偶发不可用”。

### 12.1 权限策略

权限判断分两层：

- 元数据读取链路：尽量优先使用低依赖授权方案
- 控制链路：按播放器差异决定是否需要自动化 / 辅助功能权限

这样可以尽量做到：

- 控制失败时，仍然有机会展示当前在播内容
- 权限问题被明确定位在读取还是控制阶段

### 12.2 失败类型

第一阶段必须显式处理：

- `playerNotInstalled`
- `permissionRequired`
- `launchFailed`
- `metadataUnavailable`
- `controlFailed`
- `unsupportedActivePlayer`

### 12.3 产品策略

权限不足时：

- 进入明确授权引导态
- 不静默失败
- 不只显示“当前不可控制”

控制失败时：

- 提示控制未确认
- 保留当前会话内容
- 允许用户重试

## 13. 能耗策略

因为第一阶段要求收起态显示真实播放器指示，所以关闭态不能完全停掉会话探测，但必须严格降频。

### 13.1 运行档位

第一阶段音乐模块固定为三档：

- `collapsed / closed`
  - 低频探测活跃会话
  - 只构建轻量摘要
  - 不做完整 expanded 级别状态刷新
- `expanded`
  - 中频读取活跃播放器快照
  - 支撑 UI 展示与控制反馈确认
- `sleep`
  - 暂停轮询
  - 唤醒后做一次重建

### 13.2 关闭态纪律

关闭态只允许做：

- 播放器来源探测
- `isPlaying` 判断
- 必要的最小摘要构建

关闭态不允许做：

- 高频时间推进
- 完整 expanded UI 数据组装
- 无意义封面刷新
- 控制确认以外的高频探测

### 13.3 无会话降频

当系统没有任何活跃音乐会话时：

- 关闭态探测频率进一步降低
- 避免空闲时保持不必要开销

### 13.4 控制后短时升频

当用户刚进行播放控制时：

- 允许短时间进入确认升频
- 一旦确认成功或超时失败，立刻回落

## 14. 冻结边界与允许改动

### 14.1 第一阶段允许改动

- `Modules/Music` 目录
- 音乐模块相关测试文件
- 为收起态摘要增加的最小 Shell 接线
- 必要的模块注册最小改动

### 14.2 第一阶段禁止改动

- `OverlayState`
- `NotchModuleContext`
- `ModuleEnergyPolicy`
- 多屏面板生命周期结构
- 模块自行创建完整 `NSPanel`
- 绕过 `EnergyGovernor` 单独实现后台常驻策略

如果实现收起态播放器指示时发现必须突破以上禁止项，则当前设计失效，需要回到底层线程或重新拆 scope。

---

## 15. 第一阶段验收标准

第一阶段验收按播放器逐项验证，不接受只看 UI 的“感觉可用”。

### 15.1 对 4 个首期播放器都必须验证

- 可从空态启动播放器
- 可识别当前来源
- 可读取歌名 / 歌手 / 时长
- 可执行播放 / 暂停
- 可执行上一首 / 下一首
- 收起态出现正确播放器来源与播放指示
- 展开后进入正确的播放 / 暂停内容态
- 权限缺失时进入明确引导态
- 面板关闭后不保持 expanded 级别刷新

### 15.2 对 Apple Music / Spotify 的第一阶段要求

- 不出现在空态入口
- 不写成已支持
- 若它们成为当前活跃来源，模块进入诚实降级态

## 16. 风险列表

### 16.1 适配器复杂度风险

不同播放器的控制链路可能不同，首期真正复杂的不是 UI，而是读取与控制一致性。

### 16.2 权限分裂风险

可能出现“可读但不可控”的中间状态，Spec 已要求对此提供明确 UI 与文案，不允许隐去。

### 16.3 关闭态能耗风险

收起态真实播放器指示意味着关闭态必须保留低频探测，这一处最容易踩到能耗边界。

### 16.4 多播放器仲裁风险

如果多个播放器同时有会话，缺少稳定仲裁会导致 UI 跳源或控制目标错误。

### 16.5 时间源风险

第一阶段虽然不做进度拖拽，但如果直接信任某些播放器的时间字段，仍可能出现时间显示不稳定。首期只要求“可显示”，不把时间轴纠偏纳入验收。

---

## 17. 非目标与后续阶段

下列事项明确留到后续阶段：

- Apple Music 正式接入与验证
- Spotify 正式接入与验证
- 进度拖拽
- 本地时间轴纠偏
- 更重的共享 `MusicCore` 抽象
- 更复杂的播放器能力矩阵扩展

---

## 18. Spec 总结

第一阶段音乐模块的本质不是“把占位页换成好看的 UI”，而是交付一个真实可用的音乐控制模块：

- 首期 4 个播放器真实接入
- expanded 态真实可用
- collapsed 态真实有播放指示
- 权限失败可解释
- 关闭态低频运行
- 不越过冻结边界

如果实现过程中做不到以上 6 点中的任一点，就不能宣称音乐模块第一阶段完成。
