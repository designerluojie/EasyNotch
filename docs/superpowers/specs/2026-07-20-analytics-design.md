# EasyNotch 使用埋点设计

日期：2026-07-20
状态：待评审

## 目标

回答四个问题，用于决定后续投入方向：

1. 哪些功能没人用（各模块使用占比）
2. 新用户在哪一步卡住（引导与首次权限的完成率）
3. 有多少人在用、日活多少
4. 线上出了哪些错（权限失败、控制失败等）

## 采集边界

**只上报 PV 类事件，不在 App 内生成或存储任何设备标识。**

上报的内容仅限：事件名 + 有限的枚举型属性（如设置项的取值）。

明确**不上报**：歌名、歌手、剪贴板内容、聊天内容、文件名、路径、以及任何用户输入的文本。

日活由 Umami 服务端根据 `IP + User-Agent + 每日轮换的盐` 哈希得出，App 侧不参与、也拿不到该标识。因此 App 自身不持有任何可追踪用户的信息。

## 隐私与合规

- **告知载体**：设置「关于」页最底部一行「优化改进计划」，副标题写明采集边界。官网不另设说明页（已确认）。
- **开关**：默认开启，用户可随时关闭；关闭后立即停止上报，而非下次启动生效。
- **数据流向**：Umami Cloud（第三方，服务器在境外）。免 ICP 备案。
- Umami 不使用 Cookie、不存储原始 IP，是同类方案中隐私姿态较好的一个。但服务端仍会在处理过程中看到 IP，这是"只发 PV"也无法完全规避的部分。

## UI

位置：设置 →「关于」页最底部，在「反馈问题」下方。

```
优化改进计划                                    [开关 · 默认开]
仅统计功能使用次数，不含任何个人信息与聊天内容
```

- 主标题字体与「了解我们」「反馈问题」一致，行高 36
- 副标题小一号、灰色，位于主标题下方
- 右侧为开关控件，需新建 `SettingsTogglePill`（现有 `SettingsValuePill` 是按钮样式，不适用）

## 事件清单

### 模块使用

| 事件名 | 触发时机 |
|---|---|
| `module_opened` | 模块首次在本次启动中被打开，属性 `module=music/clipboard/fileStash/pomodoro/aiChat/settings` |

### 新手引导

| 事件名 | 触发时机 |
|---|---|
| `onboarding_step` | 引导进入某一步，属性 `step=<步骤标识>` |
| `onboarding_completed` | 引导完成 |

### 设置变更

| 事件名 | 触发时机 |
|---|---|
| `setting_changed` | 用户改动某个设置项，属性 `key=<设置项>`、`value=<枚举值>` |

仅上报枚举型与开关型设置（如 `animationMode=natural`、`launchAtLogin=true`）。数值型上报取值本身（如 `clipboardMaxItems=20`）。不上报任何自由文本（如 AI API Key、自定义快捷键内容）。

### 异常

| 事件名 | 触发时机 |
|---|---|
| `permission_denied` | 权限被拒，属性 `kind=automation/accessibility` |
| `control_failed` | 播放器控制失败，属性 `player=<bundleID>` |

## 上报节流

Umami Cloud 免费额度约为每月 1 万事件。若每次打开模块都上报，几十个用户即可耗尽。

因此：**`module_opened` 每个模块在每次 App 启动中只上报一次。** 这仍能回答"哪些功能有人用、占比如何"，代价是失去"单次会话内使用了几次"的精度——对当前的决策目的（决定砍掉/投入哪个模块）足够。

`setting_changed` 与异常事件本身频次低，不做节流。

## 技术设计

### AnalyticsReporter

单一职责：把事件发往 Umami。

```
func track(_ event: AnalyticsEvent)
```

- 内部维护后台队列，**不阻塞主线程**
- 读取开关状态，关闭时直接丢弃事件
- 维护"本次启动已上报过的模块"集合，实现节流
- POST 到 Umami `/api/send`，请求体为 Umami 约定的 JSON 结构

### 失败处理

**失败一律静默丢弃，不重试、不落盘、不向用户暴露任何错误。**

埋点是次要功能，任何情况下都不得影响主体验。网络不通、Umami 故障、额度耗尽，App 表现应与没有埋点时完全一致。

### 配置

Umami 的 `websiteId` 与接口地址通过构建配置注入，便于在不接入真实服务时本地留空（留空即整体禁用上报）。

### 与现有代码的关系

- 开关状态存入现有 `settings.json`，复用 `SettingsViewModel` 的持久化机制，与其他设置项一致
- 事件触发点分散在各模块，通过依赖注入拿到 reporter，避免全局单例导致测试困难

## 测试策略

- `AnalyticsReporter` 注入可替换的传输层（`AnalyticsTransport` 协议），单测断言：
  - 开关关闭时不产生任何发送
  - `module_opened` 在同一次启动中对同一模块只发一次，不同模块各发一次
  - 事件负载不含被排除的字段
  - 传输层抛错时 `track` 不抛出、不崩溃
- 设置开关的持久化沿用现有 settings 测试模式
- 不对 Umami 真实接口做联网测试

## 明确不做（YAGNI）

- 不做用户 ID、不做设备指纹、不做跨会话追踪
- 不做留存/同期群分析（Umami 每日轮换盐，本就做不到）
- 不做事件本地缓存与补报（失败即丢弃）
- 不做实时看板（用 Umami 自带的）
- 官网不加隐私说明页（已确认）
