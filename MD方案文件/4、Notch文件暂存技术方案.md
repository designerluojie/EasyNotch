# Notch 文件暂存技术方案

日期：2026-05-05

## 1. 目标

为刘海屏工具箱中的文件暂存模块确定一条可落地的原生 macOS 技术方案，并同时满足以下产品与工程原则：

- 支持单个或多个文件 / 文件夹从 Finder、桌面、下载文件夹等位置拖入面板
- 支持从暂存列表再次拖出到 Finder、聊天工具、浏览器等外部应用
- 支持删除暂存项
- 支持应用重启后恢复暂存项
- 第一版保持“引用式暂存”，不复制大文件内容
- 关闭态保持接近无感知资源占用，不引入无必要轮询或后台刷新
- 方案必须适配 notch shell / 顶部锚点区 / 多屏单 active panel 的整体架构

一句话定位：

`文件暂存是顶部效率入口中的“临时中转托盘”，不是长期文件仓库。`

---

## 2. 技术结论

当前建议正式采用：

`引用模式（bookmark-based stash）`

即：

- 不复制拖入文件的实际内容
- 不创建独立的“临时文件副本”
- 保存原文件的 `file URL + bookmarkData`
- 重启后通过 bookmark 恢复 URL
- 拖出时重新把原文件 URL 包装成 `NSItemProvider`

一句话总结：

`文件暂存不是“缓存文件副本”，而是“缓存可恢复的原文件引用”。`

### 2.1 正式发布前提

虽然“引用式暂存”是第一版既定路线，但正式产品要把下面这条视为发布前提，而不是可选优化：

- 如果应用启用 App Sandbox，必须验证 `security-scoped bookmark` 的权限持久化与恢复链路

也就是说：

- 当前可以先按 bookmark 路线开发
- 但在没有完成 sandbox 正式验证前，不能把该能力表述为“正式可发布已完成”

---

## 3. 产品范围与非目标

### 3.1 第一版范围

第一版文件暂存模块需要覆盖：

- 拖入单个文件
- 拖入多个文件
- 拖入文件夹
- 暂存项展示
- 暂存项删除
- 暂存项拖出复用
- 重启恢复
- 自动清理策略接入设置体系

### 3.2 明确非目标

第一版不做：

- 文件副本仓库
- 原文件被删除后仍可继续使用
- 与原文件脱钩的长期留存
- 面板关闭后的后台缩略图刷新
- 为了展示效果引入持续 CPU / GPU 占用

---

## 4. 为什么这条路线适合当前产品

该路线和当前产品定位一致，原因如下：

- 顶部刘海工具箱强调高频、短时、低打扰操作
- 文件暂存的核心场景是“临时拖入，再快速拖给别的 App”
- 产品底层原则要求关闭态低占用，不适合第一版引入大文件复制、同步和回收体系

相比“复制文件内容到应用私有目录”的方案，这条路线更轻、更快，也更符合当前主链路：

`唤起 -> 查看 / 操作模块 -> 收起`

---

## 5. 与整体架构的关系

文件暂存模块必须服从整个刘海工具箱的底层规则：

- 同一时间只允许一个完整 active panel 实例展开
- 多屏场景下，完整面板单实例迁移，不能按屏幕复制完整文件容器
- 模块内容层与顶部几何、状态机、能耗治理职责分离
- 面板关闭后，文件暂存 UI 不刷新；仅保留必要的持久化与恢复能力

因此，文件暂存不应自行维护独立窗口、独立高频计时器或独立轮询器。

---

## 6. 交互与状态模型

### 6.1 模块状态

文件暂存建议明确为以下状态：

1. `collapsed`
2. `expanded-empty`
3. `drag-hover-import`
4. `expanded-filled`
5. `dragging-out`
6. `item-invalid`

### 6.2 各状态含义

`collapsed`

- 面板未展开
- 不刷新列表 UI
- 不做缩略图生成

`expanded-empty`

- 已展开但无暂存项
- 展示空态提示，例如“你还没有暂存的文件”

`drag-hover-import`

- 用户正将文件拖到文件模块可接收区域
- 展示明确的放入提示，例如“松手将文件放入暂存空间”
- 该状态必须由拖拽事件驱动，不允许用轮询模拟

`expanded-filled`

- 已有暂存项
- 展示横向文件卡片 / 图标 / 类型信息

`dragging-out`

- 用户从暂存卡片向外拖出
- 当前面板在拖拽期间应保持交互稳定，避免过早收起导致拖拽中断

`item-invalid`

- bookmark 无法恢复、文件被删除、路径失效或权限失效
- 条目保留可见，但以失效态呈现，并允许用户移除

### 6.3 能耗原则

这套状态机必须遵守：

- 事件驱动优先
- 不做持续高频监听
- 不做关闭态后台动画
- 不做关闭态后台预览刷新

换言之，状态变化应只来自：

- 面板展开 / 收起
- drag enter / leave / drop
- 用户开始拖出 / 结束拖出
- 启动恢复
- 用户删除条目

## 7. 技术方案

### 7.1 拖入

面板只接收 `UTType.fileURL` 拖放：

- 从 Finder 或桌面拖入时，系统会通过 `NSItemProvider` 提供文件 URL
- 应用从 provider 中解析出文件 URL
- 第一版支持文件与文件夹
- 第一版支持一次拖入多个项目

处理原则：

- 仅在用户完成 drop 时解析 URL
- 不在 hover 阶段预读取文件内容
- 不在导入时复制实体文件

### 7.2 保存

对拖入的文件 URL 执行 bookmark 保存：

- 开发阶段可先以普通 bookmark 验证链路
- 正式产品若启用 App Sandbox，应切换为适配 sandbox 的 `security-scoped bookmark`

持久化字段不应只保留最小 bookmark，还应保留满足 UI 展示所需的基础元数据：

- `id`
- `displayName`
- `bookmarkData`
- `itemKind`：文件 / 文件夹
- `typeLabel`：如 `PNG` / `PDF` / `Markdown` / `文件夹`
- `addedAt`
- `lastResolvedPath`（可选，用于调试或失效提示）

说明：

- 这些字段都属于轻量元数据
- 它们不会显著增加能耗
- 但能避免恢复失败后 UI 完全失去上下文

### 7.3 恢复

应用启动后：

- 从本地存储读出 bookmark 列表
- 使用 `URL(resolvingBookmarkData:...)` 恢复 URL
- 检查 bookmark 是否 stale
- 必要时刷新 bookmark
- 检查文件是否仍可访问

恢复结果分两类：

- 可恢复：进入正常暂存项
- 不可恢复：进入失效项状态，而不是静默消失

原因：

- 对用户来说，“文件不见了”和“文件失效了”是两种完全不同的体验
- 第一版虽然不做副本模式，但仍应保留用户可理解的失效反馈

### 7.4 拖出

从暂存列表拖出时：

- 不生成副本文件
- 直接将恢复出来的原始 URL 再次包装为 `NSItemProvider`
- 目标应用接收到的仍然是原文件

但需要区分两种表述：

- `已验证能力`
- `目标支持范围`

正式文档中不应把所有外部应用都写成已稳定验证。更准确的口径应为：

- 目标支持 Finder、聊天工具、浏览器等常见拖放目标
- 具体适配效果按验证结果区分“已验证”与“待验证”

### 7.5 展示信息与缩略图策略

Figma 设计要求文件卡片至少体现：

- 文件名
- 类型
- 图标或缩略图

为兼顾还原度与能耗，建议采用轻量策略：

- 文件夹与未知类型：优先展示系统图标
- 图片 / PDF 等可预览类型：仅在面板可见、条目进入可见区时按需生成小尺寸缩略图
- 缩略图不在关闭态后台刷新
- 缩略图失败时回退到系统图标

这意味着：

- 缩略图生成是“按需可选增强”
- 不是常驻后台任务
- 不应为了缩略图引入持续刷新或预热

---

## 8. 列表规则与清理策略

### 8.1 排序与去重

建议规则：

- 按最近加入时间排序
- 重复拖入同一路径的项目默认去重
- 若产品后续需要“重新置顶”，可在重复拖入时刷新排序，但第一版可先不做复杂策略

### 8.2 容器承载方式

由于 notch 面板横向空间有限：

- 文件列表应以横向卡片容器为主
- 超出可见宽度后，采用横向滚动或裁切后滚动的承载方式
- 不建议在第一版扩展为高列表或多行瀑布

### 8.3 自动清理

设置体系已要求支持文件暂存自动清理，因此技术方案必须预留：

- 手动清空
- 单项删除
- 自动清理策略

自动清理触发可以是：

- 按保留时长
- 按数量上限
- 或二者组合

第一版文档至少应明确：

- 不允许无限堆积
- 清理动作应是低频、事件驱动或启动时触发
- 不做常驻高频巡检

---

## 9. 已知边界与产品行为

### 9.1 原文件依赖

因为当前方案保存的是原文件引用，所以如果原文件后来：

- 被删除
- 被移动
- 权限失效

那么暂存项可能无法再次拖出。

### 9.2 失效项处理

当暂存项失效时，建议产品行为为：

- 保留条目可见
- 显示失效态样式或提示
- 允许用户手动移除
- 不自动尝试后台无限恢复

这样可以兼顾：

- 用户可理解性
- 关闭态低占用
- 代码复杂度可控

### 9.3 不是“文件副本”

当前方案不保证：

- 原文件离线后仍可用
- 文件被移动后仍完全无感恢复
- 文件被删后还能继续拖出

它更适合：

- 临时托盘
- 短时中转
- 当前工作流中的快速拖放

---

## 10. 与“复制模式”的区别

另一种可选路线是：

`复制模式`

即拖入后把文件复制到应用自己的 stash 目录。

### 10.1 复制模式优点

- 原文件被删掉后，副本仍然可用
- 更像真正的“文件仓库”

### 10.2 复制模式缺点

- 占用磁盘空间
- 大文件拖入慢
- 需要清理策略
- 需要处理重复文件、命名冲突、文件更新同步等问题
- 更容易违背当前“关闭态低占用、最小完整实现”的优先级

### 10.3 当前建议

第一版不采用复制模式，先采用：

`引用模式作为默认技术路线`

后续仅在产品明确需要“长期留存”或“原文件脱钩”时，再评估是否增加复制模式。

- Finder / 桌面文件可拖入

- 暂存项可再次拖出到外部应用

- 还没有覆盖多屏迁移 / 全屏 / 合盖 / 菜单栏自动隐藏等场景

因此，Demo 是技术方向佐证，不是正式完成证明。

## 12. 最终推荐结论

正式建议：

- 第一版继续采用 `引用模式（bookmark-based stash）`
- 保持“不复制文件内容”的产品决策不变
- 正式发布前必须补齐 sandbox bookmark 验证
- 拖拽交互必须按事件驱动接入 notch / 顶部锚点状态机
- 列表规则、失效项策略、自动清理策略必须进入正式方案，而不是留空

一句话结论：

`文件暂存第一版应做成“低能耗的原文件引用托盘”，而不是“高成本的副本仓库”。`
