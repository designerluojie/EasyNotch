# Notch 番茄钟技术方案

日期：2026-05-05

## 1. 目标

为刘海屏工具箱中的番茄钟模块确定一条可落地的 macOS 原生技术方案，并同时满足以下产品与工程原则：

- 支持 `25 / 45 / 60` 分钟专注时长选择
- 支持开始专注、暂停专注、停止专注
- 展示今日累计专注时长
- 面板关闭后不维持无意义高频 UI 刷新，但计时核心仍能继续推进
- 适配 notch shell / 顶部锚点区 / 多屏单 active panel 的整体架构
- 保持低打扰，不把番茄钟做成高噪音、高常驻成本的独立计时器应用

一句话定位：

`番茄钟是顶部入口中的“低打扰专注状态机”，不是常驻复杂任务管理器。`

---

## 2. 技术结论

当前建议正式采用：

`应用级番茄钟核心常驻 + 刘海面板按需展示 + 基于绝对结束时间的倒计时`

即：

- 番茄钟核心作为应用级单例存在
- 计时真值不依赖面板是否展开
- UI 只在可见时做 1Hz 倒计时刷新
- 面板不可见时，不保留秒级 UI timer，只保留必要的完成调度与状态恢复
- 当前轮次使用 `targetEndAt` 驱动，而不是用持续累加秒数作为唯一真值

一句话总结：

`番茄钟不是“靠一直跑 timer 才能活着”的 UI，而是“有独立计时核心、由刘海入口按需投影出来”的模块。`

---

## 3. 产品范围与非目标

### 3.1 第一版范围

第一版番茄钟模块需要覆盖：

- 选择 `25 / 45 / 60` 分钟专注时长
- 开始专注
- 暂停专注
- 停止专注
- 专注完成提示
- 休息准备页
- 停止休息
- 休息完成提示
- 今日累计专注时长展示
- 重启后恢复当前轮次

### 3.2 按当前设计稿补充的产品结论

结合飞书 PRD 与当前 Figma 节点，第一版还应明确：

- 专注完成后进入一轮休息流
- 当前设计稿中的休息时长固定为 `5:00`
- 今日累计只统计“专注时长”，不统计休息时长
- 完成提示采用 notch 短提示，不直接强制展开完整大面板

### 3.3 明确非目标

第一版不做：

- 任务清单绑定
- 多任务番茄钟并行
- 系统级通知中心深度编排
- 历史专注报表与周/月统计
- 自定义专注时长与自定义休息时长
- 常驻呼吸动画、流光动画、后台秒级列表刷新

---

## 4. 为什么这条路线适合当前产品

这条路线和当前产品定位一致，原因如下：

- 刘海工具箱强调“唤起快、收起快、关闭态低占用”
- 番茄钟的真需求是“状态连续”，不是“面板必须一直开着”
- 用户在专注过程中大概率不会持续盯着完整面板，而是偶尔看一眼顶部状态
- 多屏与全屏场景下，如果把计时逻辑绑在某个具体面板实例上，状态同步和恢复都容易出错

相比“完整 UI 常驻 + 一直 1Hz 驱动整块面板”的路线，这条方案更符合当前主链路：

`开始专注 -> 顶部低打扰查看 -> 完成提示 -> 继续下一轮或收起`

---

## 5. 与整体架构的关系

番茄钟模块必须服从整个刘海工具箱的底层规则：

- 同一时间只允许一个完整 active panel 实例展开
- 多屏场景下，完整面板单实例迁移，不能按屏幕复制完整番茄钟容器
- 模块内容层与顶部几何、状态机、能耗治理职责分离
- 面板关闭后，番茄钟不刷新完整 UI；仅保留必要计时核心

因此，番茄钟不应自行维护：

- 独立窗口
- 独立高频常驻动画
- 与其他模块割裂的全局状态容器

---

## 6. 交互与状态模型

### 6.1 设计稿已明确的可见状态

当前 Figma 节点已经明确给出以下状态：

1. `collapsed-focus-running`
2. `expanded-focus-idle`
3. `expanded-focus-running`
4. `expanded-break-ready`
5. `toast-focus-finished`
6. `toast-break-finished`

### 6.2 各状态含义

`collapsed-focus-running`

- notch 高度为 `34`
- 左侧展示“专注中”与图标
- 右侧展示剩余时间，如 `24:59`
- 属于低打扰进行中摘要，不是完整控制面板

`expanded-focus-idle`

- notch 展开为 `257`
- 中央展示大号时间，如 `25:00`
- 主按钮为“开始专注”
- 下方有 `25 / 45 / 60` 选项
- 底部展示“今日已累计专注 X 分钟”

`expanded-focus-running`

- 展开高度仍为 `257`
- 大号时间进入倒计时
- 主按钮切为“暂停”
- 下方出现“停止专注”
- 计时器外围的完整背景圆圈切换为环形进度条
- 环形进度条按“已消耗专注时长 / 本轮专注总时长”持续推进
- 当环形进度条达到 `100%` 时，视为本轮专注完成

`expanded-break-ready`

- 专注完成后进入休息准备页
- 中央展示 `5:00`
- 主按钮为“开始休息”
- 下方操作为“停止休息”
- 底部继续展示今日累计专注时长

`toast-focus-finished`

- notch 高度为 `99`
- 中央展示勾选图标
- 文案为“完成！准备休息一下吧”

`toast-break-finished`

- notch 高度为 `99`
- 中央展示勾选图标
- 文案为“休息完成，可以继续进入专注了！”

### 6.3 逻辑上必需但当前设计稿未单独出图的状态

为了满足 PRD 中“暂停专注”的要求，以及完成休息链路，技术上还必须补齐以下逻辑态：

1. `expanded-focus-paused`
2. `expanded-break-running`
3. `expanded-break-paused`

当前建议：

- `expanded-focus-paused` 复用专注大卡骨架，主按钮改为“继续专注”，保留当前剩余时间
- `expanded-break-running` 复用休息页骨架，主按钮改为“暂停”，继续显示倒计时
- `expanded-break-paused` 复用休息页骨架，主按钮改为“继续休息”

但必须明确：

- 上述三个状态属于实现必需态
- 当前不是设计稿中已经明确锁定的视觉稿
- 实现前应由产品 / 设计补图，或确认允许先按同构骨架复用

### 6.4 今日累计时长规则

根据当前设计稿中的动态数字变化，建议把“今日累计专注时长”定义为：

`今日已完成专注秒数 + 当前专注轮次已消耗秒数（仅在专注进行中计入）`

这意味着：

- 专注进行中，底部数字应实时增长
- 暂停时，累计值冻结
- 停止专注时，已消耗的专注时长保留计入今日累计
- 休息阶段不增加今日累计

## 7. 默认交互链路

### 7.1 开始专注链路

1. 用户打开刘海面板
2. 切换到番茄钟模块
3. 选择 `25 / 45 / 60`
4. 点击“开始专注”
5. 进入 `expanded-focus-running`
6. 用户收起面板后，若当前设计允许摘要展示，则进入 `collapsed-focus-running`

### 7.2 暂停 / 恢复链路

1. 用户在专注进行中点击“暂停”
2. 当前轮次冻结剩余时间
3. 今日累计停止增长
4. 用户点击“继续专注”后恢复倒计时

### 7.3 停止专注链路

1. 用户在专注进行中点击“停止专注”
2. 当前轮次结束
3. 已产生的专注累计保留
4. 回到 `expanded-focus-idle`
5. 默认保留用户最近一次选择的专注时长

### 7.4 专注完成链路

1. 倒计时到达 `0`
2. 当前专注轮次正式完成
3. 今日累计增加本轮完整专注时长
4. notch 切换到 `toast-focus-finished`
5. 短提示结束后进入 `expanded-break-ready` 或等待用户再次展开查看

### 7.5 休息链路

1. 用户点击“开始休息”
2. 进入 `expanded-break-running`
3. 休息过程中不增加今日累计
4. 用户可暂停或停止休息
5. 休息完成后展示 `toast-break-finished`
6. 返回下一轮专注待开始状态

## 8. 技术方案

### 8.1 计时真值

当前建议不要依赖“每秒递减一个整数”的本地变量作为唯一真值，而是保存：

- `phase`
- `status`
- `selectedFocusDuration`
- `breakDuration`
- `startedAt`
- `targetEndAt`
- `remainingWhenPaused`

计时规则：

- 运行态：`remaining = targetEndAt - now`
- 暂停态：`remaining = remainingWhenPaused`
- 完成态：由 `remaining <= 0` 驱动相位切换

这样做的价值：

- 面板关闭后无需维护完整秒级 UI 状态
- 应用重启后可以恢复
- 睡眠 / 唤醒后可以直接根据绝对时间校正

### 8.2 为什么不用“永远常驻 1Hz 全局 Timer”

原因不是做不到，而是不适合当前产品：

- 关闭态会带来无意义刷新
- 多屏和窗口迁移时更容易出现多处同时刷新
- 与当前产品的低能耗纪律冲突

一句话原则：

`时间状态要常驻，秒级 UI 刷新不要常驻。`

### 8.3 进度环计算规则

专注进行中时，计时器外围背景圆圈不再只是静态装饰，而应切换为当前轮次的环形进度条。

建议进度值定义为：

`progress = elapsedFocusSeconds / totalFocusSeconds`

其中：

- `elapsedFocusSeconds = totalFocusSeconds - remainingFocusSeconds`
- `progress` 取值范围为 `0...1`
- UI 层按该值驱动环形描边长度或 trim 比例

完成判定规则：

- 当 `progress >= 1`
- 或 `remainingFocusSeconds <= 0`

即视为本轮专注结束，并触发：

1. 当前专注轮次完成结算
2. notch 切换到 `toast-focus-finished`
3. 后续进入休息准备流

### 8.4 可见时刷新策略

建议拆成两层：

- `PomodoroCore`
  - 负责轮次、时间、状态切换、持久化
- `PomodoroPresenter`
  - 只在刘海摘要或完整面板可见时，附加 1Hz UI 刷新

具体规则：

- 完整面板可见：允许 1Hz 刷新大号倒计时与累计时长
- 收起摘要可见：只刷新小号剩余时间
- 面板与摘要都不可见：不保留 1Hz 重复 timer

### 8.5 相位切换

建议相位定义为：

- `focus`
- `break`

状态定义为：

- `idle`
- `running`
- `paused`
- `finishedToast`

约束：

- 同一时间只允许一个全局活动番茄钟轮次
- 不允许同时存在“专注进行中”和“休息进行中”

## 9. 数据模型建议

### 9.1 当前轮次快照

`PomodoroSessionSnapshot`

建议字段：

- `id`
- `phase`
- `status`
- `selectedFocusDurationSeconds`
- `breakDurationSeconds`
- `startedAt`
- `targetEndAt`
- `remainingWhenPaused`
- `lastUpdatedAt`

说明：

- 这是恢复当前轮次的核心快照
- 不需要每秒落盘
- 只需在关键状态切换时更新

### 9.2 当日累计模型

`PomodoroDailyStats`

建议字段：

- `dayKey`
- `focusedSecondsCompleted`
- `lastSessionId`

说明：

- 已完成专注秒数持久化保存
- 当前运行中的“增量部分”在展示层动态计算
- 到新的一天后按本地自然日切换 `dayKey`

### 9.3 本地存储原则

继续采用：

`local-first 轻量持久化`

即：

- 小体量 JSON 或偏好存储即可
- 不引入数据库
- 不做云同步

## 10. 持久化与恢复策略

### 10.1 何时持久化

建议在以下时机持久化：

- 开始专注
- 暂停
- 恢复
- 停止
- 专注完成
- 开始休息
- 停止休息
- 休息完成
- 应用即将退出或进入后台时

### 10.2 重启恢复

应用启动后：

1. 读取 `PomodoroSessionSnapshot`
2. 判断是否存在未结束轮次
3. 根据 `phase + status + targetEndAt` 计算当前应处于的真实状态
4. 如已超时，则直接推进到对应完成提示或下一状态

示例：

- 若应用重启时专注轮次已超时，应直接进入 `toast-focus-finished`
- 若应用重启时休息轮次已超时，应直接进入 `toast-break-finished`
- 若应用重启时为暂停态，应按暂停态恢复，不自动继续

### 10.3 睡眠 / 唤醒

对于 macOS 睡眠 / 唤醒：

- 睡眠期间计时仍视为自然流逝
- 唤醒后通过 `targetEndAt - now` 一次性校正
- 不依赖“睡眠前漏掉的若干次 timer tick”

## 11. UI 展示方案

### 11.1 展开态

根据当前设计稿，展开态至少需要展示：

- 大号倒计时
- 主操作按钮
- 进度环
- 次操作按钮
- `25 / 45 / 60` 时长选项
- 今日累计专注时长

### 11.2 收起态

当前设计稿已明确专注进行中的收起态：

- 左侧为状态标签“专注中”
- 右侧为剩余时间

但要注意：

- 当前节点没有单独给出“休息进行中”的收起态
- 如果实现期需要该状态，建议先复用同构布局，并把文案替换为“休息中”

### 11.3 完成提示态

完成提示建议由宿主层统一控制：

- 从当前高度平滑过渡到 `99`
- 展示勾选图标与文案
- 提示结束后再进入下一逻辑态

这样可以避免：

- 直接弹完整大面板造成打扰
- 与“低打扰顶部入口”定位冲突

## 12. 模块划分建议

### 12.1 宿主层

由刘海屏工具箱统一提供：

- `DisplayTopologyService`

- `AnchorGeometryCalculator`
- `OverlayCoordinator`
- `PanelWindowController`
- `InteractionStateMachine`
- `HotzoneController`
- `ContentHost`
- `EnergyGovernor`

### 12.2 番茄钟核心层

建议独立为应用级模块：

1. `PomodoroCore`
2. `PomodoroSessionStore`
3. `PomodoroStateReducer`
4. `PomodoroRecoveryResolver`
5. `PomodoroCompletionCoordinator`

### 12.3 番茄钟展示层

只在模块可见时工作：

1. `PomodoroViewModel`
2. `PomodoroModuleView`
3. `PomodoroCollapsedIndicator`
4. 完成提示视图

一句话边界：

`核心层保证时间正确推进，展示层只负责把当前状态投影到 notch UI。`

## 13. 能耗策略

### 13.1 核心原则

`时间连续，刷新克制，关闭休眠`

### 13.2 必须具备的节能措施

- 不在面板关闭且摘要不可见时保留 1Hz UI timer
- 不在后台跑持续动画
- 不按秒持久化到磁盘
- 完成检查优先依赖 `targetEndAt` 与生命周期事件恢复
- 仅在当前活跃屏幕的可见入口上刷新摘要时间

### 13.3 与 EnergyGovernor 协同

建议把番茄钟纳入统一能耗治理：

- 完整面板关闭时暂停大卡 UI 刷新
- 切出番茄钟模块时暂停其展开态秒级刷新
- 睡眠前停止纯展示型 timer
- 唤醒后按当前状态最小代价恢复

---

## 14. 多屏与窗口行为

### 14.1 多屏策略

- 番茄钟会话是应用级全局单例
- 不为每块屏维护独立番茄钟副本
- 完整面板仍然遵守单 active panel 迁移规则

### 14.2 进行中的摘要归属

建议默认策略：

- 只有当前 active screen 的顶部入口展示运行摘要
- 其他屏幕保留轻量唤起入口，但不复制完整进行中倒计时摘要

原因：

- 避免多个屏同时呈现“像是多个活动实例”
- 保持单实例心智一致

### 14.3 特殊场景

以下场景都必须验证倒计时连续性和 UI 归位正确性：

- 外接屏插拔
- 合盖模式
- 全屏 / 非全屏切换
- 菜单栏自动隐藏 / 不隐藏
- 睡眠 / 唤醒

---

## 15. 已知边界与待确认项

### 15.1 当前设计稿缺口

当前 Figma 节点未单独提供：

- 暂停专注态视觉稿
- 进行中休息态视觉稿
- 暂停休息态视觉稿
- 休息进行中的收起摘要态

因此技术方案需要先把这些作为“逻辑必需态”写清楚，但不能把它们表述为“设计已完全定稿”。

### 15.2 系统通知不是当前主链路

当前设计稿明确的是 notch 内提示，不是系统通知中心提示。

第一版建议：

- 先保证 notch 内状态提示闭环
- 不把系统通知当作主完成路径

### 15.3 第一版不引入复杂统计体系

当前“今日累计专注时长”已经足够支持主场景。

第一版不建议同步引入：

- 专注次数统计
- 周报
- 月报
- 连续打卡

---

## 16. 最终推荐结论

正式建议：

- 当前默认采用 `应用级番茄钟核心常驻 + 刘海面板按需展示 + targetEndAt 驱动倒计时`
- 专注时长固定支持 `25 / 45 / 60`
- 按当前设计稿补入固定 `5` 分钟休息流
- 今日累计专注时长采用“已完成专注 + 当前专注进行中增量”的计算方式
- 必须补齐暂停专注、进行中休息、暂停休息等逻辑态，并与设计确认其最终视觉稿
- 面板关闭后停止无意义秒级 UI 刷新，只保留必要计时核心与完成恢复能力

一句话结论：

`番茄钟第一版应做成“应用级连续计时核心 + 刘海入口低打扰投影”，而不是“依赖完整面板常驻刷新的复杂计时器 UI”。`
