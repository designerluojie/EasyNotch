# Notch 剪贴板技术方案

日期：2026-05-05

## 1. 目标

为刘海屏工具箱中的剪贴板模块确定一条可落地的 macOS 原生技术方案，优先满足以下目标：

- 支持系统级剪贴板历史采集
- 第一版支持纯文字、富文本、图片、标准 SVG、Figma 图形、Figma 文字、文件 / 多文件 / 文件夹
- 支持在刘海屏面板中快速回看最近复制内容
- 支持点击历史项后重新写回系统剪贴板
- 保持关闭态近乎无感知资源占用
- 保持架构简单，降低多屏、休眠恢复、面板状态切换带来的 Bug 风险

---

## 2. 技术结论

当前建议正式采用：

`应用级剪贴板核心常驻采集 + 刘海面板按需展示`

即：

- 剪贴板监听核心作为应用级单例存在
- 不依赖面板是否打开
- 面板只负责展示、选择和再次写回
- 面板关闭后停止不必要的 UI 刷新，但不停止历史采集

一句话总结：

`剪贴板能力不是常驻 UI，而是常驻低频核心 + 按需展开的刘海模块页。`

---

## 3. 当前默认方案

### 3.1 核心原则

`采集常驻，展示休眠`

用户在任意应用中复制内容后：

1. 应用级 `ClipboardMonitor` 监听到系统剪贴板变化
2. 解析不同类型内容为统一的 `ClipboardCapture`
3. 立即进入本地历史存储
4. 用户打开刘海面板并进入剪贴板模块时，再按需读取并展示历史
5. 用户点击某条历史项时，将内容重新写回系统剪贴板

### 3.2 为什么采用这条路线

优点：

- 保留关闭面板期间的复制历史，不会漏采集
- 避免为剪贴板单独维护一个悬浮窗或多实例窗口
- 与 PRD 中“同一时间只允许一个完整面板 active”一致
- 采集与 UI 解耦，更容易控制能耗和刷新频率
- 复用现有剪贴板技术逻辑，迁移成本最低

相比“面板打开时才启动监听”的路线，这条方案更符合真实使用场景。相比“独立剪贴板浮窗”的路线，这条方案更稳，更容易压低窗口层级和状态同步问题。

## 4. 适用范围

### 4.1 第一版支持内容

- `plainText`
- `richText`
- `image`
- `svg`
- `figmaScene`
- `file`

对应能力：

- 普通文本
- 代码片段
- URL 文本
- 富文本
- 标准 SVG
- 截图
- 网页复制的图片
- 一般位图图片内容
- Figma 普通复制图形
- Figma 普通复制文字
- 单文件
- 多文件
- 文件夹

### 4.2 第一版不纳入范围

以下内容不作为当前稳定支持范围：

- 云同步
- 跨设备文件实体同步
- 把任意 Figma 普通复制自动当作标准 SVG 处理
- 非必要的后台持续动画和高频 UI 刷新

说明：

当前 App 的剪贴板模块目标是“在 notch 工具箱中稳定承载完整首版剪贴板历史能力”，但仍然不把它做成云同步产品或复杂的跨设备资产系统。

## 5. 与刘海屏宿主的集成方式

### 5.1 模块定位

剪贴板不是独立面板，而是刘海屏工具箱中的一个内嵌模块页：

- 统一挂在顶部 notch shell / handler 所承载的主面板中
- 与音乐、文件暂存、AI Chat、番茄钟共享同一个 `NSPanel` 宿主
- 顶部导航、面板展开收起、动效节奏由宿主层统一管理

### 5.2 多屏策略

- 每块屏幕都可以有轻量入口
- 同一时间只允许一个完整面板展开
- 剪贴板历史是应用级全局共享数据
- 不为每块屏维护单独的剪贴板副本

### 5.3 面板状态原则

- 面板关闭：剪贴板核心继续低频采集，UI 不刷新
- 面板展开但当前不在剪贴板模块：只保留数据存在，不挂载高成本列表刷新
- 用户切到剪贴板模块：按需加载历史列表、缩略图和时间文案

## 6. 监听方案

### 6.1 监听入口

使用：

`NSPasteboard.general`

作为系统剪贴板读取入口。

### 6.2 监听方式

当前正式建议继续采用：

`changeCount` 轮询

实现方式：

- 使用 `NSPasteboard.general.changeCount`
- 使用低频 `Timer` 比较计数变化
- 默认轮询间隔为 `0.5s`

### 6.3 为什么不用事件订阅

原因不是“做不到”，而是“不值得为了理想化事件流引入更脆弱的链路”：

- macOS 没有适合当前场景的稳定通用剪贴板变化事件流
- 剪贴板类产品普遍采用轮询
- `changeCount` 方案简单、稳定、可控

一句话原则：

`这里优先选择现实稳定方案，而不是概念上更优雅但不稳的方案。`

---

## 7. 低能耗策略

### 7.1 核心原则

`低频监听，按需展示，关闭休眠`

### 7.2 必须具备的节能措施

- 轮询频率节流，默认 `0.5s`
- 系统睡眠前暂停监听
- 屏幕睡眠前暂停监听
- 唤醒后恢复监听并重建基线
- 面板关闭后不做额外 UI 高频刷新
- 未进入剪贴板模块时，不做无意义列表更新和缩略图重复解码
- 富文本预览、SVG 缩略图、Figma 预览文案、文件图标计算都采用懒处理

### 7.3 EnergyGovernor 协同

建议把剪贴板也纳入宿主的 `EnergyGovernor` 统一管理：

- 面板关闭时暂停 UI 级 timer
- 模块不可见时暂停“相对时间文案”刷新
- 缩略图生成改为懒加载
- 清理任务使用低频调度，不做持续轮询

说明：

关闭态并不是“完全不运行任何任务”，而是只保留必要的低频核心任务。对剪贴板来说，这个必要任务就是 `changeCount` 监听。

## 8. 去重与防递归策略

### 8.1 应用自身写回去重

当用户点击某条历史项后，应用会把内容重新写回系统剪贴板。

这一步必须避免被自己再次采集。

建议策略：

- 在 `PasteExecutor` 写回前生成一次本次写回标记
- `ClipboardMonitor` 监听到变化时先判断是否命中“本应用刚写回”的内容
- 命中则忽略本次采集

### 8.2 存储层去重

除了防递归，还要处理普通重复复制问题。

建议保留：

`contentHash + contentType`

作为历史去重键。

命中重复内容时：

- 不新增新条目
- 更新时间
- 移到最前

### 8.3 去重价值

- 降低历史噪音
- 降低磁盘写入次数
- 避免用户反复点击后制造大量重复项

## 9. 数据模型

### 9.1 统一采集模型

`ClipboardCapture`

建议字段：

- `contentType`
- `previewText`
- `contentHash`
- `copiedAt`
- `sourceAppBundleId`
- `sourceAppName`
- `payloadData`
- `payloadExtension`

说明：

这是监听层和存储层之间的桥梁模型。

### 9.2 历史项模型

`ClipboardHistoryItem`

建议字段：

- `id`
- `contentType`
- `previewText`
- `contentHash`
- `copiedAt`
- `sourceAppBundleId`
- `sourceAppName`
- `payloadPath`
- `thumbnailPath`

说明：

- `thumbnailPath` 只对图片类型启用，可选
- `thumbnailPath` 也可用于 SVG 或特定 Figma 图形卡片的预览缓存
- 如果第一版希望进一步简化，也可以先不落独立缩略图文件，而是在展示时直接从 payload 懒生成

### 9.3 设置项

建议增加：

- `maxClipboardItems`
- `clipboardAutoCleanupPolicy`

其中：

- `maxClipboardItems` 对应：`5 / 10 / 15 / 20 / 30 / 50`
- `clipboardAutoCleanupPolicy` 对应：不自动 / 每日 / 每周 / 每月

### 9.4 文件 payload

`ClipboardFilePayload`

建议字段：

- `files: [ClipboardFileReference]`

单个文件引用建议字段：

- `originalPath`
- `fileName`
- `isDirectory`
- `bookmarkData`

说明：

- 单文件、多文件、文件夹统一走 `file` 内容类型
- 第一版文件类型采用引用式保存，不复制文件实体

### 9.5 Figma payload

`ClipboardFigmaPayload`

建议职责：

- 保存 Figma 私有 scene / HTML representation
- 区分 Figma 图形与 Figma 文字
- 为历史卡片生成预览文案
- 再次粘贴时恢复原始 representation

## 10. 存储方案

### 10.1 存储原则

继续采用：

`本地文件持久化（local-first）`

即：

- 元数据保存在本地 JSON
- payload 单独落盘
- 不引入数据库
- 不做云同步

### 10.2 推荐目录

建议目录形态：

- `~/Library/Application Support/<AppName>/Clipboard/history.json`
- `~/Library/Application Support/<AppName>/Clipboard/Payloads/`

### 10.3 文本存储

- 文本直接保存 UTF-8 数据
- 预览文案使用原始文本或截断文本

### 10.4 富文本存储

- 优先读取并保存 `RTF`
- 如果只有 `HTML`，先转成富文本，再转成 `RTF`
- 列表中只显示纯文本预览
- 再次粘贴时回写 `RTF`

### 10.5 图片存储

- 图片统一转成 PNG payload
- 保留单张大小上限
- 超限时不入库，或根据产品策略只保留摘要不保留原图

一句话原则：

`图片统一归一化，避免多格式分支把展示和回贴链路做复杂。`

### 10.6 标准 SVG 存储

- 识别 `public.svg-image`、`public.svg`、`image/svg+xml`
- 保存原始 SVG 文本
- 列表中可显示 SVG 预览或缩略图
- 再次粘贴时回写标准 SVG 数据

说明：

- 只有真正进入系统剪贴板的标准 SVG 才走这一类
- Figma 普通复制图形不默认归到标准 SVG

### 10.7 Figma 图形 / Figma 文字存储

- 保存 Figma 私有 scene / HTML representation
- 图形项默认显示“Figma 图形”或通用 Figma 卡片
- 文字项优先显示实际文字内容
- 目标是可以稳定贴回 Figma

说明：

- 这类内容重点是回贴到 Figma
- 不保证贴到其他应用时保持通用矢量语义

### 10.8 文件 / 多文件 / 文件夹存储

- 不复制文件内容本身
- 保存原始文件引用
- 通过 `bookmarkData` 持久化文件 URL
- 再次粘贴时还原 URL 并写回系统剪贴板

说明：

- 如果文件被删除、移动、磁盘卸载，历史项可能失效
- 这套方案适合单机本地历史，不适合直接做跨设备同步

## 11. UI 展示方案

### 11.1 展示内容

根据本期 PRD，剪贴板模块列表中每条记录建议展示：

- 来源图标
- 复制时间
- 内容摘要
- 图片缩略图
- SVG 预览或缩略图
- Figma 图形卡片 / Figma 文字内容
- 文件 / 文件夹图标与名称摘要

### 11.2 展示形态

结合现有视觉稿，建议采用：

- 顶部为统一模块导航
- 中部为剪贴板列表区
- 默认展示最近项
- 空状态展示“你还没有剪贴板内容”之类的文案

### 11.3 刷新纪律

- 面板关闭后停止列表刷新
- 不做后台持续动画
- 相对时间文案不做秒级刷新
- 图片、SVG、Figma 图形卡片缩略图只在进入模块时按需生成或读取

## 13. 用户操作链路

### 13.1 采集链路

1. 用户在任意应用复制文本、富文本、图片、标准 SVG、Figma 内容或文件
2. `ClipboardMonitor` 检测到 `changeCount` 变化
3. `ClipboardNormalizer` 识别内容类型并生成 `ClipboardCapture`
4. `ClipboardStore` 去重后持久化
5. 如果剪贴板模块当前可见，则触发最小必要 UI 更新

### 13.2 再次使用链路

1. 用户打开刘海面板
2. 用户切换到剪贴板模块
3. 读取最近历史并展示
4. 用户点击某条记录
5. `PasteExecutor` 将内容重新写回系统剪贴板

### 13.3 自动粘贴策略

第一版默认建议：

`只保证重新写回剪贴板，不默认自动执行 Cmd + V`

原因：

- 自动粘贴依赖辅助功能权限
- 会显著增加前台焦点、权限、系统事件发送的复杂度
- 不利于初版稳定性控制

后续如果产品确认要做，再作为增强能力单独引入。

## 14. 清理策略

### 14.1 条数上限

按设置项控制历史上限：

- `5`
- `10`
- `15`
- `20`
- `30`
- `50`

超过上限时：

- 删除最旧历史项
- 同步删除对应 payload 文件

### 14.2 自动清理

支持：

- 不自动
- 每日
- 每周
- 每月

建议实现方式：

- 使用低频调度或在应用启动 / 唤醒时顺带检查
- 不使用高频 timer 持续盯住时间点

## 15. 已知边界

### 15.1 第一版不是完整剪贴板博物馆

当前目标是 notch 工具箱中的高频实用模块，不是完整的桌面级云同步剪贴板产品。

因此当前不追求：

- 跨设备同步
- 文件实体跨设备同步
- 把 Figma 普通复制统一转换为标准 SVG

### 15.2 来源应用识别存在降级

来源图标依赖复制发生时的前台应用识别。

如果识别失败：

- 不阻塞入库
- 降级为通用图标

### 15.3 文件类型仍然是引用式存储

文件 / 多文件 / 文件夹当前保存的是 bookmark 引用，不是副本。

因此如果原文件：

- 被删除
- 被移动
- 权限失效

则该历史项可能无法再次使用。

### 15.4 图片内容需要大小控制

如果用户连续复制大图：

- 会推高磁盘占用
- 也会提高缩略图生成成本

所以必须有图片大小限制和自动清理策略。

## 16. 当前推荐结论

正式建议：

- 当前默认采用 `应用级剪贴板核心常驻采集 + 刘海面板按需展示`
- 第一版支持纯文字、富文本、图片、标准 SVG、Figma 图形、Figma 文字、文件 / 多文件 / 文件夹
- 监听继续采用 `NSPasteboard.changeCount` 轮询
- 必须配套：
  - 频率节流
  - 睡眠 / 唤醒暂停恢复
  - 应用自身写回递归抑制
  - `contentHash + contentType` 去重
  - 面板关闭后的 UI 休眠

一句话结论：

`剪贴板先做成“低频常驻采集的应用级核心”，再做成“刘海面板中的按需使用模块”，不要一开始就做成常驻复杂 UI 或全格式管理器。`
