# 刘海屏工具产品文档

# 一、产品概述

## 1.1 项目背景

MacBook 刘海区具备高可达、低路径成本、与系统顶部状态区邻近的交互优势。当前项目的目标不是单纯把刘海做成一个视觉玩具，而是做成一个可常驻、低打扰、适配多设备形态的顶部工具箱。

## 1.2 项目目标

- 把 Mac 顶部刘海区或顶部中心入口做成一个高频可用的效率工具箱。
- 在有刘海和无刘海设备上都保持自然的入口体验。
- 通过默认模块和基础工具组合，让用户在顶部入口完成音乐控制、文件中转、AI 对话、剪贴板回看、专注计时等高频操作。

## 1.3 产品原则/核心决策

- 功能范围以最终功能清单为唯一准绳。
- 有刘海内置屏使用完整 notch shell；无刘海设备或外接无刘海屏退化为顶部中心 handler 或模拟刘海入口。
- 每块屏幕都可以有唤起入口，但同一时间只允许一个完整面板展开。
- 面板关闭后，模块展示层进入休眠，不保留无意义刷新；但像剪贴板采集、文件引用持久化、番茄钟计时这类必要后台核心链路按最低能耗策略继续运行。
- AI Chat 仅支持用户自配 API，不提供默认模型和默认服务。
- 音乐模块允许多播放器接入，但要区分“已验证”与“待验证”能力，避免过度承诺。

## 1.4 工程要求

- 不依赖写死的刘海尺寸，统一基于顶部锚点区动态布局。
- 支持多屏、外接屏、合盖、全屏、睡眠唤醒等复杂场景。
- 保持关闭态近乎无感知资源占用。仅保留必要的低频核心任务，如剪贴板 changeCount 监听、文件暂存恢复能力、专注计时持久化。
- 同一时间只允许一个完整面板实例处于 active 状态。

## 1.5 核心使用场景

1. 用户在工作过程中通过顶部入口快速展开面板，控制正在播放的音乐。
2. 用户把文件临时拖入刘海面板暂存文件，随后再拖给聊天工具、浏览器或 Finder。
3. 用户快速回看最近复制过的文字或图片，并再次使用。
4. 用户通过顶部面板发起 AI 对话，或把图片丢给已配置的模型处理。
5. 用户直接在顶部入口开启一轮番茄钟，并查看当天累计专注时长。
6. 用户在设置中调整功能顺序、清理策略、动效表现和全局快捷键。

# 二、功能模块清单

## 2.1 新手引导

- 首次启动展示引导。
- 默认只展示一次。
- 重点说明顶部入口形态、核心模块、全局快捷键和 AI 配置入口。

## 2.2 音乐控制

- 作为默认模块存在。
- 未播放时提供播放器启动入口。
- 启动目标包含：
  - Apple Music
  - 网易云音乐
  - QQ 音乐
  - 酷狗音乐
  - 汽水音乐
  - Spotify
- 播放中：
  - 未展开时展示播放源 Logo 与动态播放提示
  - 展开后展示封面、歌曲名、歌手、播放进度
  - 支持播放、暂停、上一首、下一首
- 当前技术验证已覆盖：
  - QQ 音乐
  - 网易云音乐
  - 酷狗音乐
  - 汽水音乐
- Apple Music 与 Spotify 属于当前产品目标范围，但仍需在实现阶段补充接入验证。

## 2.3 文件暂存

- 支持单个、多个文件或文件夹拖入面板。
- 支持从暂存列表再次拖出到外部应用。
- 支持重启后恢复暂存项。
- 第一版默认采用“引用式暂存”，即保存原文件引用，不复制大文件内容。

## 2.4 AI Chat

- 未配置 API 时引导用户配置。
- 支持的提供商：
  - DeepSeek
  - Qwen
  - ChatGPT
  - Gemini
- 已配置 API 后支持：
  - 文字对话
  - 图片输入
  - 历史对话保留
  - 模型选择
- 当所选模型不支持图片时，必须给出明确提示。

## 2.5 剪贴板

- 展示来源图标、复制时间、内容摘要。
- 第一版支持纯文字、富文本、图片、标准SVG、Figma图形、Figma文字、文件/多文件/文件夹。
- 最大保存条数由设置控制。
- 需要支持自动清理策略。
- 剪贴板历史为应用级全局共享，不按屏幕分别维护。
- 点击历史项默认执行“写回系统剪贴板”；自动粘贴不属于第一版默认能力。
- 面板关闭时停止剪贴板列表 UI 刷新，但不停止历史采集。

## 2.6 番茄钟

- 支持开始专注、中途暂停、专注主动停止、专注完成、5分钟休息。
- 支持 `25 / 45 / 60` 分钟时长选择。
- 展示今日累计专注时长。

## 2.7 设置

- 用设置：
  - 登录时打开
  - 全局快捷键设置，默认 `Command + Option + T`
  - 模拟 Mac 刘海屏，仅对非刘海屏设备生效
  - 展开动效效果：自然 / Q弹
  - 展开动效速度：正常 / 快 / 慢
- 功能选项：
  - 功能展示顺序排序
  - 文件暂存自动清理：不自动 / 每日 / 每周 / 每月
  - 剪贴板最大保存数：`5 / 10 / 15 / 20 / 30 / 50`
  - 剪贴板自动清理：不自动 / 每日 / 每周 / 每月
  - AI 提供商配置与移除入口
- 关于我们：
  - 应用介绍
  - Logo 展示
  - 官网跳转

# 三、技术方案

## 3.1 底层架构分层

为了保证多屏适配、形态退化和低能耗纪律，采用以下底层模块拆分：

- `DisplayTopologyService`
  - 监听 `NSScreen.screens`
  - 处理插拔显示器、分辨率变化、缩放和屏幕参数变化
- `ScreenProfileResolver`
  - 区分有刘海内置屏、无刘海内置屏、外接无刘海屏
  - 输出当前屏幕的显示策略
- `AnchorGeometryCalculator`
  - 基于 `safeAreaInsets.top`、`auxiliaryTopLeftArea`、`auxiliaryTopRightArea` 计算顶部锚点区
  - 输出 idle、hoverHint、expanded 等状态所需几何参数
- `OverlayCoordinator`
  - 协调每块屏幕的轻量入口与唯一 active panel
  - 处理多屏迁移、合盖、全屏隐藏与恢复
- `PanelWindowController`
  - 基于 `NSPanel` 管理边框、层级、透明、非激活与动画几何变化
- `InteractionStateMachine`
  - 统一 `idle -> hoverHint -> expanded -> collapsing` 状态链
- `HotzoneController`
  - 管理顶部窄热区与命中策略
- `ContentHost`
  - 承载各功能模块的挂载、卸载与懒加载
- `EnergyGovernor`
  - 统一动画、计时器、轮询、刷新、订阅的休眠与恢复策略
- `ClipboardCore`
  - 应用级单例，负责监听、归一化、去重、持久化、清理，不依赖面板是否展开

## 3.2 音乐控制技术方案

- 信息读取优先复用 `MediaRemote / Now Playing` 能力。
- 在第三方播放器时间源不稳定时，采用“本地时间轴为主，系统值只做校准”的策略。
- 当前已有人工验证的控制能力包括：
  - 进度拖拽
  - 暂停
  - 开始
  - 上一首
  - 下一首
- 对于尚未完整验证的播放器，不在文档中承诺“稳定支持”，而写为“目标接入”。

## 3.3 文件暂存技术方案

- 第一版采用 `bookmarkData` 引用式暂存。
- 核心链路：
  - 拖入解析 `fileURL`
  - 将原始 URL 转为 bookmark
  - 本地持久化
  - 重启恢复 URL
  - 拖出时重新封装为 `NSItemProvider`
- 第一版不做副本模式，不做大文件复制。

## 3.4 剪贴板技术方案

- 使用 `NSPasteboard` 作为系统剪贴板读取入口。
- 第一版支持纯文字、富文本、图片、标准SVG、Figma图形、Figma文字、文件/多文件/文件夹。
- 由于系统层不存在理想事件流，可采用 `changeCount` 轮询作为现实方案，但必须配套：
  - 频率节流
  - 睡眠 / 唤醒暂停恢复
  - 应用自身写回去重
  - 面板关闭后不做额外 UI 高频刷新

## 3.5 AI Chat 技术方案

- 每个提供商独立配置与移除。
- 模型能力需要声明是否支持图片输入。
- 对话历史、模型选择和未配置态引导需要共存，但不能让配置态复杂度污染未配置态。
- API 密钥必须安全存储，不得明文散落在普通偏好配置中。

## 3.6 番茄钟技术方案

- 计时逻辑与 UI 刷新解耦。
- 面板关闭后不维持高频动画刷新。
- 今日累计时长需要独立持久化。

# 四、关键交互与状态模型

## 4.1 顶部入口交互

- `idle`：默认关闭态，只保留顶部入口。
- `hoverHint`：用户进入热区后提供轻量反馈。
- `expanded`：用户主动点击展开，显示完整模块内容。
- `collapsing`：用户离开1000ms后回收内容并返回关闭态。

## 4.2 多屏交互

- 每块屏幕可以有轻量入口。
- 只有一个完整面板实例可以展开。
- 当用户在另一块屏上触发时，当前面板先收起，再迁移到目标屏。

## 4.3 模块装载策略

- 默认模块和其他模块由 `ContentHost` 管理。
- 不可见模块不应持续刷新。
- 当前未激活屏幕的完整内容不应常驻装载。

# 五、 多屏 / 合盖 / 全屏 / 无刘海等边界场景

## 5.1 设备与屏幕形态

- 有刘海内置屏：使用完整 notch shell。
- 无刘海内置屏：退化为顶部中心 handler 或模拟刘海入口。
- 外接无刘海屏：支持轻量入口；是否展示完整面板由单实例策略决定。

## 5.2 系统状态场景

- 多屏同时存在
- 主副屏切换
- 合盖模式
- 分辨率与缩放比例变化
- 菜单栏自动隐藏 / 不隐藏
- 全屏 / 非全屏切换
- 睡眠 / 唤醒

## 5.3 边界规则

- 全屏场景下不能错误显示或抢占系统焦点。
- 合盖后要重新绑定当前主交互屏。
- 外接屏数量增加后，不能线性增加完整面板实例和高频资源开销。

## 5.4 能耗与性能约束

- 关闭态需接近无感知资源占用。
- 动画只在用户主动触发的状态切换时执行。
- 不做待机呼吸动效、持续流光、持续毛玻璃重绘。
- 模块不可见时：
  - 不刷新 AI 内容
  - 不高频刷新音乐详情 UI
  - 不高频刷新番茄钟界面
  - 不保留无意义计时器
- 对必须轮询的能力，要明确节流和休眠策略。

## 5.5 开发边界 / 非目标

- 不做跨平台版本。
- 不写死具体机型刘海尺寸。
- 不做同时展开多个完整面板实例。
- 不把旧文档里出现但不在功能清单内的模块纳入当前版本。
- 不提供默认 AI 模型或默认 API 服务。
- 不为了视觉效果牺牲明显的空闲态能耗。
- 文件暂存第一版不做副本仓库。

# 六、风险与待验证项

## 6.1 已知风险

- 第三方播放器时间源可能不稳定，导致进度条与真实播放状态偏移。
- 外接屏、合盖、全屏和菜单栏状态变化会显著增加窗口协调复杂度。
- 剪贴板能力可能需要轮询，需要严控资源成本。
- 文件暂存采用引用模式后，原文件被删除或移动会导致拖出失败。

## 6.2 待验证项

- Apple Music 与 Spotify 的完整控制和进度同步验证。
- 非刘海设备模拟刘海入口的视觉与交互边界。
- App Sandbox 场景下 bookmark 持久化权限策略。
- AI 图片输入在不同模型能力矩阵下的交互提示一致性。

## 6.3 成功标准

- 有刘海与无刘海设备都能自然工作。
- 多屏、全屏、合盖、唤醒后不乱跳、不误触。
- 同一时间只有一个完整 active panel。
- 音乐控制、文件暂存、AI Chat、剪贴板、番茄钟五条主链路可用。
- 设置项能覆盖当前功能清单要求的关键控制项。
- 关闭态资源占用极低。
- 不依赖任何写死设备尺寸。
